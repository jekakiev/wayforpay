## Signature rules

Use WayForPay HMAC-MD5 signatures exactly in the documented field order and UTF-8 encoding.

### Purchase request

Doc: https://wiki.wayforpay.com/en/view/852102

Base string:

`merchantAccount;merchantDomainName;orderReference;orderDate;amount;currency;productName[0..n];productCount[0..n];productPrice[0..n]`

Notes:

- Array values are appended in order, not JSON-encoded.
- Product arrays must be aligned by index.
- A wrong array order is a common cause of signature mismatch.

### Callback request from WayForPay

Doc: https://wiki.wayforpay.com/en/view/852102

Base string:

`merchantAccount;orderReference;amount;currency;authCode;cardPan;transactionStatus;reasonCode`

Notes:

- Verify the signature before any order mutation.
- Keep callback verification code isolated and testable.

### Callback acknowledgment from merchant

Doc: https://wiki.wayforpay.com/en/view/852102

Base string:

`orderReference;status;time`

Notes:

- `status` is normally `accept` in a successful acknowledgment.
- `time` should be generated by the merchant at response time.

### `CHECK_STATUS` request

Doc: https://wiki.wayforpay.com/en/view/852117

Base string:

`merchantAccount;orderReference`

### `REFUND` request

Doc: https://wiki.wayforpay.com/en/view/852115

Base string:

`merchantAccount;orderReference;amount;currency`

### `REFUND` response from WayForPay

Doc: https://wiki.wayforpay.com/en/view/852115

Base string:

`merchantAccount;orderReference;transactionStatus;reasonCode`

### `SETTLE` request and response

Doc: https://wiki.wayforpay.com/en/view/852113

Request base string:

`merchantAccount;orderReference;amount;currency`

Response base string:

`merchantAccount;orderReference;transactionStatus;reasonCode`

## Status handling

Official status doc: https://wiki.wayforpay.com/en/view/852131

Most important statuses for merchant logic:

| Status | Meaning | Merchant action |
| --- | --- | --- |
| `Approved` | Payment succeeded | Mark paid after verified callback or successful reconciliation |
| `Pending` | Under antifraud review | Keep order pending and reconcile later |
| `inProcessing` | Still being processed | Keep order pending and retry status check later |
| `WaitingAuthComplete` | Hold succeeded, waiting for capture | Keep authorized state and use `SETTLE` or `REFUND` if appropriate |
| `Declined` | Payment failed or was cancelled | Mark failed and keep retry UX/business rules separate |
| `Expired` | Payment window expired | Mark expired or cancelled |
| `Refunded/Voided` | Funds returned or hold released | Mark refunded or voided |
| `RefundInProcessing` | Refund accepted but not completed | Keep refund pending and reconcile later |

## Merchant-side state guidance

Recommended high-level mapping:

- `Approved` -> paid/completed
- `Pending`, `inProcessing` -> pending_payment
- `WaitingAuthComplete` -> authorized_awaiting_capture
- `Declined`, `Expired` -> failed_or_cancelled
- `Refunded/Voided` -> refunded_or_voided
- `RefundInProcessing` -> refund_pending

Practical rules:

- Distinguish gateway status from business fulfillment status.
- Do not ship or activate paid-only access on `Pending` or `inProcessing`.
- Treat `WaitingAuthComplete` as a separate state from normal success.
- Reconciliation logic should be safe to run multiple times.

## Terminal vs non-terminal statuses

Treat these as non-terminal and eligible for scheduled reconciliation:

- `Pending`
- `inProcessing`
- `RefundInProcessing`
- `WaitingAuthComplete` if your flow still expects capture or void

Treat these as terminal for most merchant flows:

- `Approved`
- `Declined`
- `Expired`
- `Refunded/Voided`

If the product uses delayed capture, `WaitingAuthComplete` is terminal for authorization but not terminal for full settlement.

## Common failure patterns

Check these first during review/debug work:

- semicolon-separated fields built in the wrong order
- amount or currency normalized differently than the gateway expects
- cart arrays reordered before signature generation
- signature comparison done after order mutation instead of before
- callback and `CHECK_STATUS` racing each other without idempotency protection
- `Pending` or `inProcessing` treated as success
- `WaitingAuthComplete` treated as a final paid state
- refund response accepted without validating returned status and signature
